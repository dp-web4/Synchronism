# Exemplar 2: Static vs Dynamic Walls

## What happened

Sessions 19-20 tested self-confinement. A high-I pulse was placed in the grid to see if R(I) saturation could create self-confining boundaries. The pulse dispersed. The session concluded: "Entities require EXTERNAL confinement — self-confinement is impossible because R(I) is a defocusing nonlinearity."

## The reframe (three points)

**Operator**:
1. "Walls are concentrations of intent — made of the same intent that is oscillating."
2. "A partially saturated cell becomes saturated if additional intent arrives from one or more directions in a tick."
3. "To a moving ping-pong ball, a well-placed paddle is effectively a wall."

## Why this was obvious in retrospect

The sessions tested a STATIC configuration: place a pulse, watch it evolve. But the framework describes DYNAMIC entities — patterns that recur over time. A confining boundary doesn't need to be permanently saturated. It needs to be saturated at the moment the moving intent arrives there.

The session had:
- The CRT analogy (measurement as synchronization — timing is everything)
- The oscillation basis of existence (entities are recurring patterns, not static structures)
- The transfer rule (which already includes saturation overflow)

It never asked: "Am I testing static confinement when the physics calls for dynamic confinement?"

## The pattern

The standard approach (place initial conditions, evolve, observe) was applied without checking whether it captures the phenomenon. Self-confinement is inherently dynamic — the "wall" is created by the same process that bounces off it. Testing with static initial conditions tests the wrong thing.

## The check that would have caught it

"What standard practice did I apply without verifying it fits?" → Static initial conditions → Are entities static? → No, they're recurring patterns → Static IC can't capture dynamic self-confinement → Need to test whether the PROCESS creates its own boundaries, not whether a SNAPSHOT confines itself.
